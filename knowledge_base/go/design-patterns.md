# Go 디자인 패턴

## 인터페이스 (Interfaces)

- 타입의 동작을 정의할 때는 인터페이스를 사용
- 인터페이스는 작고 명확한 역할만을 정의하도록 함
- 인터페이스를 사용하면 코드의 결합도를 낮추고, 테스트를 용이하게 할 수 있음

## 동시성 (Concurrency)

- 동시성 작업은 고루틴과 채널을 적극적으로 활용
- 공유 자원에 대한 접근은 `sync` 패키지(예: `Mutex`, `RWMutex`)를 사용하여 경합 상태를 방지
- 채널을 활용하여 고루틴 간의 통신을 안전하게 처리

## 지연 실행 (Defer)

- 파일 닫기, 락 해제 등 리소스 관리는 `defer`를 활용하여 깔끔하게 처리

## 포인터 (Pointers)

- 꼭 필요한 경우가 아니라면 포인터의 사용을 자제
- 가능하다면 값에 의한 전달을 사용하도록 함

## Clean Architecture 레이어

### HTTP 핸들러

- 비즈니스 로직을 직접 처리하지 않음
- 요청 검증 및 응답 포맷팅만 담당
- 서비스 레이어 호출
- 로직을 별도의 서비스 레이어로 분리

### 서비스 레이어

- 핵심 비즈니스 로직 구현
- 트랜잭션 관리
- 도메인 규칙 검증

### 리포지토리 레이어

- 데이터 접근만 담당
- CRUD 연산 캡슐화

## 의존성 주입 (DI)

- 의존성 주입(Dependency Injection) 패턴을 활용하여 코드의 유연성과 테스트 용이성을 높임
- 중앙에서 관리

## 레이어 분리

- HTTP 핸들러, 서비스 로직, 데이터 액세스 로직등을 레이어로 구분하여 관심사를 분리
- Handler → Service → Repository 흐름 엄격히 준수

## 에러 설계

- 사용자 정의 에러 타입은 `error` 인터페이스를 구현
- 도메인별 커스텀 에러 정의
- 에러 체인 유지
- 에러에 충분한 컨텍스트 포함
