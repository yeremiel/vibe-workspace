# ワークスペース構造の設計

人間とAIエージェント（Claude Code）が**同じドキュメントを読み、同じルールで働く**ように設計したローカルワークスペースの構造。プロジェクトが増えるにつれて標準が厚みを増していく、生きたシステムを目指している。

## ディレクトリ構造

```
~/workspace/
├── knowledge_base/    # プロジェクト横断の開発標準
├── obsidian/          # 作業ログ、意思決定記録、レビュー（Obsidian Vault）
├── projects/
│   └── self/          # 個人プロジェクト集
└── CLAUDE.md          # ワークスペースルートのAIコンテキスト
```

## 設計原則

### knowledge_base/ — 「人間とAIが共に見る開発標準」

言語別・ドメイン別の開発標準を一箇所にまとめて管理する。
すべてのプロジェクトのCLAUDE.mdから`@`参照で読み込む。

- `common/`: APIレスポンス標準、Gitワークフローなど言語に依存しないルール
- `go/`: Goのコーディング規約、デザインパターン
- `flutter/`: Flutter/Dartの規約、gotchas

この構造のおかげで、新しいプロジェクトを始める際にCLAUDE.mdで標準ドキュメントを参照するだけで、AIエージェントは一貫したスタイルでコードを生成し、新メンバー（人間）は同じドキュメントでオンボーディングできる。

**「生きている」ことが重要だ。** 標準は一度作って終わりではなく、実プロジェクトで見つけたパターンや失敗が再び標準に反映される。たとえば `flutter/gotchas.md` は作業中に遭遇した落とし穴の蓄積であり、`go/design-patterns.md` は複数のバックエンドを経て洗練された構造判断だ。新しい言語/ドメインが加われば `knowledge_base/` 配下に新しいフォルダが増える。

### obsidian/ — 「作業記録ハブ」

プロジェクト進行中の意思決定、試行錯誤の記録、PhaseごとのレビューをObsidian Vaultで管理する。
コードリポジトリに入れるのが難しい「なぜこうしたか」を残す場所だ。

### projects/self/ — 「実験場」

外部の要件なしに自ら企画・開発する個人プロジェクト集。
各プロジェクトにCLAUDE.mdを置いて、プロジェクト固有のAIコンテキストを管理する。

## CLAUDE.mdの階層構造

```
workspace/CLAUDE.md                          # ワークスペース共通コンテキスト（作業者/環境/禁止事項）
└── projects/self/<project>/CLAUDE.md        # プロジェクト固有コンテキスト（スタック/アーキ/例外）
    └── @knowledge_base/go/conventions.md    # 標準ドキュメント参照（言語/ドメイン別）
```

AIエージェントは現在の作業ディレクトリのCLAUDE.mdを読み込み、その中で参照しているknowledge_baseのドキュメントも合わせてロードする。

この階層は**人間とAIが同じエントリポイントを持つ**ように設計されている。新メンバーをオンボーディングするときに見せるドキュメントと、AIエージェントが作業開始時に読むドキュメントが完全に同じだ。したがって「人間にはREADME、AIには別プロンプト」のような二重管理は不要になる。
