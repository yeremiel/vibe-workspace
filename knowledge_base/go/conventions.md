# Go 코딩 컨벤션

## 명명 규칙

**패키지 이름 (Package Names)**:
- 모든 패키지 이름은 소문자로 작성
- 단일 단어를 사용하는 것을 선호
- 여러 단어를 사용해야 할 경우, 단어를 붙여서 사용 (예: `userprofile`, `productcatalog`)
- 언더스코어(`_`)는 사용하지 않음
- 명확하고 간결한 이름을 사용
- `util`, `common`, `helper` 와 같은 일반적인 이름은 사용을 피함
- **예시**: `users`, `products`, `orders`, `database`, `auth`

**변수 이름 (Variable Names)**:
- 지역 변수(함수 내에서만 사용되는 변수)는 `camelCase`를 사용
- 공개된(exported) 변수는 `PascalCase`를 사용
- 짧은 스코프에서는 짧은 이름을 사용할 수 있음 (예: `i`, `j`, `k`)
- 넓은 스코프에서는 명확한 의미를 전달하는 설명적인 이름을 사용
- **예시**: `userProfile`, `totalCount`, `UserName`, `UserProfileList`

**함수 이름 (Function Names)**:
- 비공개 함수는 `camelCase`를 사용
- 공개된(exported) 함수는 `PascalCase`를 사용
- 함수 이름은 동사로 시작하여 수행하는 작업을 명확히 설명
- Boolean을 반환하는 함수는 `Is`, `Has`, `Can` 등의 접두사를 사용
- **예시**: `getUser`, `createUser`, `CalculateTotal`, `IsValidUser`

**상수 (Constants)**:
- 비공개(unexported) 상수는 `camelCase`를 사용
- 공개된(exported) 상수는 `PascalCase`를 사용
- Go는 `UPPER_SNAKE_CASE`를 상수 네이밍에 사용하지 않는다 (C/Python 관행과 다름)
- **예시**: `maxUsers`, `defaultTimeout`, `MaxRetries`, `DefaultPageSize`

## 에러 처리

- 모든 에러는 항상 처리해야 함
- 에러를 반환할 수 있는 함수를 호출한 후에는 항상 `if err != nil`로 에러를 확인
- 에러는 가능한 한 빨리 반환
- 에러에 문맥 정보를 추가하기 위해 `fmt.Errorf` 또는 `errors.Wrap`을 활용
- 에러는 디버깅에 도움이 되는 문맥 정보와 함께 반환

## 함수 작성

- 함수는 단일 책임을 갖도록 짧게 유지
- 가능하면 20-30줄을 넘지 않도록 함
- 함수가 너무 길어지면, 여러 개의 작은 보조 함수로 분리
- 함수는 하나의 책임만 가져야 함
- **예외**: 프로토콜 핸드셰이크, 파일 유효성 검사 등 단계가 본질적으로 많은 로직은 30줄을 초과할 수 있다. 이 경우 각 단계가 주석이나 코드 구조로 명확히 구분되어야 하며, 가능한 보조 함수 추출을 먼저 시도해야 한다

## 주석 (Comments)

- 복잡하거나 명확하지 않은 코드에는 주석을 작성
- 공개된(exported) 함수와 타입에 대한 설명은 주석으로 문서화
- 주석은 코드가 "무엇"을 하는지 보다는 "왜" 그렇게 했는지를 설명
- 주석은 코드의 의도를 설명해야 함

## Import 정리

`import` 문은 다음 세 그룹으로 분류하고, 각 그룹은 알파벳 순으로 정렬:
1. 표준 라이브러리 패키지
2. 외부 패키지
3. 내부 패키지

## 파일 구성 (File Organization)

- 각 파일은 하나의 공개 구조체나 인터페이스만 포함되도록 함
- 파일이 잘 분리되어야 모듈화가 잘 되어서 유지보수가 편함

## 테스트

- 공개된 모든 함수에 대해 단위 테스트를 작성
- 테스트 케이스 이름은 명확하고, 엣지 케이스를 커버하도록 작성
- 테이블 기반 테스트 패턴 사용
- Mock은 `test/mocks/`에 배치
