# Git 브랜치 & 커밋 규칙

## 브랜치 전략

작업의 성격에 따라 브랜치를 분리한다.

- 기능 개발: `feature/<기능명>`
- 버그 수정: `fix/<버그명>`
- 문서 작업: `docs/<문서명>`
- 리팩토링: `refactor/<대상>`
- 테스트 작업: `test/<대상>`
- 기타 작업: `chore/<작업명>`

기본 원칙:
- 하나의 브랜치는 하나의 목적만 가진다
- main 브랜치에서 직접 작업하지 않는다
- 작업 완료 후 main 브랜치로 병합한다

---

## 브랜치 생성 기준

모든 변경이 반드시 새로운 브랜치를 필요로 하지는 않는다.

다음 경우에만 브랜치를 생성한다:
- 하나의 독립된 기능 단위 작업
- 사용자 동작 또는 API/화면에 영향을 주는 버그 수정
- 여러 파일에 걸친 구조적 변경
- 리뷰 또는 롤백 가능성이 필요한 변경

다음 경우에는 기존 브랜치 또는 main 브랜치에서 작업할 수 있다:
- 단순 오탈자 수정
- 로그 메시지 수정
- 주석 또는 문서의 경미한 수정
- 코드 포맷 정리 (기능 변경 없음)

단, 브랜치를 분리하지 않더라도 커밋 메시지 규칙은 동일하게 적용한다.

---

## 단기 브랜치 운영 원칙

브랜치는 작업 중에만 유지되는 임시 단위로 취급한다.

- 병합이 완료된 브랜치는 즉시 삭제한다
- 24시간 이상 유지되는 브랜치는 작업 단위를 재검토하거나 분리한다
- 목적이 불명확해진 브랜치는 정리 후 재생성한다

브랜치 이름은 작업 목적을 설명하는 용도로만 사용하며,
장기적인 변경 이력 관리는 커밋 로그를 기준으로 한다.

---

## 커밋 메시지 컨벤션

커밋 메시지는 다음 형식을 따른다.

```
<type>(<scope>): <subject>
```

### type (필수)

- `feat`      : 새로운 기능 추가
- `fix`       : 버그 수정
- `docs`      : 문서 추가 또는 수정
- `refactor`  : 기능 변경 없는 코드 구조 개선
- `test`      : 테스트 코드 추가 또는 수정
- `chore`     : 설정, 빌드, 의존성 등 기타 작업
- `perf`      : 성능 개선

### scope (권장)

변경이 발생한 범위를 간단히 명시한다.
프로젝트별 scope 목록은 각 프로젝트의 CLAUDE.md를 참고한다.

### subject (필수)

- **한국어로 작성**
- 현재형 동사 사용
- 마침표 사용 금지
- 50자 이내
- 변경 내용을 명확히 표현

예시:
```
feat(auth): 로그인 API 추가
fix(auth): 만료된 JWT 토큰 처리
refactor(repository): MongoDB 쿼리 로직 분리
docs(api): 응답 형식 업데이트
```

---

## Breaking Change

기존 동작과 호환되지 않는 변경이 있을 경우 `!`를 사용한다.

```
feat(auth)!: 토큰 구조 변경
```

## 브랜치 이름과 커밋 메시지 규칙 연계

브랜치 이름은 커밋 메시지의 `type`과 `scope`를 결정하는 기준으로 사용한다.

### 브랜치 → type 매핑

- `feature/*`   → `feat`
- `fix/*`       → `fix`
- `docs/*`      → `docs`
- `refactor/*`  → `refactor`
- `test/*`      → `test`
- `chore/*`     → `chore`

### 브랜치 → scope 규칙

- 브랜치 이름에서 첫 번째 의미 단어를 scope로 사용
- 하이픈(`-`)으로 연결된 경우 첫 단어만 scope로 사용

예시:
- `feature/auth-login`         → `feat(auth)`
- `fix/user-duplicate-email`   → `fix(user)`
- `docs/api-response`          → `docs(api)`
- `refactor/repository-mongo`  → `refactor(repository)`

## 커밋 메시지 생성 규칙 (Claude Code)

Claude Code는 커밋을 수행하기 전에 반드시 커밋 메시지를 먼저 제안해야 한다.

제안 시 다음 정보를 포함한다:
- 작업 유형(type)
- 변경 범위(scope)
- 커밋 메시지(subject)

형식 예시:
```
feat(auth): 로그인 API 추가
```

사용자가 승인하거나 수정 요청을 한 경우에만 실제 커밋을 진행한다.
